# Skill: 单篇研究型文献精读

## Goal
对单篇研究论文进行高质量精读，输出的重点是：
1. 研究问题是否成立；
2. 作者的核心设想（Hypothesis，假设）是什么；
3. 证据链是否足以支撑结论；
4. 真正的创新点、边界条件和可疑之处在哪里。

本 skill 的目标是“拆解 + 重构”：
- 拆解：把论文拆成问题、假设、方法、证据、结论；
- 重构：用研究者语言把它重新组织成一条清楚、紧凑、可判断的逻辑链。

## Use this skill when
满足以下任一情况时启用：
- 用户要求精读一篇研究型论文；
- 用户要求总结单篇论文的研究逻辑；
- 用户要求分析论文创新点、实验设计、证据链或文章质量；
- 用户要做组会汇报、journal club、论文笔记；
- 用户上传了 1 篇研究论文，希望得到深度解读。

## Do not use this skill when
以下情况不优先使用本 skill：
- 用户只是要摘要、翻译、改写摘要；
- 用户要做多篇文献比较或综述；
- 用户上传的是纯综述、评论、观点文章，且没有要求按研究论文方式拆解；
- 用户只要提取结论、图表、实验条件等局部信息。

## Required inputs
尽量从论文中提取以下信息：
- 标题
- 研究对象 / 材料体系 / 器件体系
- 研究问题
- 核心策略
- 关键实验或关键图
- 主要结论
- 对照组与基线
- 任何可能涉及边界条件的信息

如果输入不完整，也要先输出“当前可判断版本”，同时明确缺口。

## Core rules
1. 不按摘要、引言、结果、结论的原顺序机械复述。
2. 先抓主线，再看细节。
3. 区分四类证据：核心实验、交叉印证、求新炫技、Show 工作量。
4. 不把“图很多”误判为“逻辑很强”。
5. 任何结论尽量对应具体证据。
6. 对“真问题”保持判断：这到底是领域瓶颈，还是包装后的伪命题？
7. 对“性能提升”保持判断：比较是否公平，提升是否真实，是否带来额外代价。
8. 输出时避免空泛夸赞，不写“很有启发”“很有意义”之类套话。
9. 如证据不足，明确写“不足以判断”，不要硬补。
10. 用户没有特别要求时，默认采用研究者口吻，不写科普文风。

## Evidence classification standard
在分析实验时，强制按以下四类区分：

### 1. 核心实验
定义：去掉之后，主结论会明显失去支撑的证据。
判断问题：
- 它是否直接支撑核心主张？
- 没有它，论文是否仍能站住？

### 2. 交叉印证
定义：与核心实验形成闭环或互证的辅助证据。
判断问题：
- 它是否提高了可信度？
- 它是否解决了“孤证不成立”的问题？

### 3. 求新炫技
定义：增强新颖性、视觉冲击力或顶刊气质，但对主结论并非绝对必要的部分。
判断问题：
- 没有它，主逻辑是否仍成立？
- 它增加的是“必要证据”还是“发表气质”？

### 4. Show 工作量
定义：为系统性、参数摸索、鲁棒性、审稿应对而做的大量展开。
判断问题：
- 它是否只是加厚文章？
- 它对研究型结论的必要性有多高？
- 它是否更偏应用开发（Development，开发）价值？

## Output requirements
输出必须严格使用以下结构与标题。

### 一、论文主线（四段法，每项一句话）
- 小背景：
- 真问题：
- 课题设计：
- 验证方法：

要求：
- 小背景只写具体研究切口；
- 真问题优先写“两难”“瓶颈”“失配”“限制步骤”；
- 课题设计写核心假设，不写流程细节；
- 验证方法只写最关键的验证路径。

### 二、作者的研究故事
用 3–5 句话重构整篇论文的逻辑：
- 为什么这个问题值得做？
- 作者提出了什么设想？
- 他们拿什么证据证明这个设想？
- 结论成立到什么程度？

### 三、证据链成分分析
按四类输出：

#### 1. 核心实验
- 实验 / 数据：
- 支撑的核心结论：
- 不可替代的原因：

#### 2. 交叉印证
- 辅助证据：
- 对应哪条核心结论：
- 互证是否充分：

#### 3. 求新炫技
- 炫技部分：
- 没有它主结论是否仍成立：
- 它真正增加的价值：

#### 4. Show 工作量
- 工作量部分：
- 对核心结论的必要性：
- 对应用或工程优化的意义：

### 四、批判性判断
至少回答以下问题：
- 作者提出的问题是否成立？
- 证据链是否存在跳步？
- 各实验结果之间是否自洽？
- 对照组与比较基线是否公平？
- 性能提升是否可能来自隐藏变量？
- 是否引入了新的代价、副作用或约束？
- 最亮眼的结果去掉之后，文章是否仍然成立？

### 五、一句话定性
用一句话写出论文真正的核心创新。

要求：
- 禁止写成“制备了 XX 材料，实现了 XX 提升”；
- 必须指出新策略、新机制、新设计原则，或被打破的旧约束；
- 句子要能回答“这篇文章到底新在哪里”。

### 六、最终评价
- 逻辑严密性：强 / 中 / 弱
- 创新性：强 / 中 / 弱
- 证据完成度：高 / 中 / 低
- 可借鉴价值：高 / 中 / 低

并补两句：
- 最值得学的地方：
- 最值得怀疑的地方：

## Optional add-ons
按用户需求追加：

### A. 组会 3 分钟汇报版
输出：
- 研究背景 1 句
- 核心问题 1 句
- 核心策略 1 句
- 最关键证据 2–3 句
- 你的判断 2 句

### B. Obsidian 精简笔记版
输出：
- 标题
- 一句话总结
- 四段法
- 核心证据
- 可疑点
- 可迁移启发

## Style constraints
- 语言简洁、硬、可判断；
- 允许批评，但批评必须有依据；
- 不堆实验名词，不把表征清单当分析；
- 避免模板化抒情；
- 默认中文输出；
- 除非用户要求，否则不展开冗长背景教学。

## Failure handling
遇到以下情况时这样处理：
- 论文不完整：先做“有限精读版”，显式列出缺失信息；
- 论文质量很差：照常分析，但把“证据不足”写清楚；
- 文章更像综述：说明不适合本 skill，并切换到综述型结构；
- 用户只给摘要：只做“基于摘要的初判版”，不要假装读过全文。

## Final instruction
你输出的重点不是“把论文内容说一遍”，而是判断：
- 问题是否真；
- 假设是否清楚；
- 证据是否硬；
- 创新是否成立；
- 这篇文章到底值不值得学。
